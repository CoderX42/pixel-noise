# 像素迷宫 · 每日七关 — 详细实现提示词

将下文整体作为对 AI 或工程实现的**规格说明**。目标：浏览器内可运行的**像素风**迷宫游戏，**每日种子**生成可复现的**七关递进迷宫**，记录**最少步数**并维护**排行榜**（优先本地，可扩展服务端）。

**业务逻辑扩写版（用户旅程、状态机、数据模型）见：** [BUSINESS-LOGIC.md](../pixel-maze/BUSINESS-LOGIC.md)  
**给 Cursor / Trae 生成单页原型的可复制提示词见：** [pixel-maze-ide-prototype.md](pixel-maze-ide-prototype.md)

---

## 一、产品定义

### 1.1 一句话

**「每日迷宫」**：以**自然日（用户本地时区）**为种子，生成 **7 关**由易到难的**程序化迷宫**；玩家从起点走到终点，**步数越少越好**；成绩进入**当日排行榜**（及可选的历史最佳）。

### 1.2 核心循环

1. 进入页面 → 显示**今日日期**与「开始今日挑战」。  
2. 依次通过 **关卡 1 → 7**；每关**新迷宫**（同日内固定，换日则变）。  
3. 每关达成：**记录本关步数**；七关全部完成后汇总 **总步数**。  
4. 若总步数优于本人/本地榜记录，**写入排行榜**（可输入昵称，默认匿名）。  
5. 提供 **「重玩今日」**（重置七关进度但迷宫布局不变）与 **「查看解法步数」**（可选：显示理论最短步数，用于对比）。

### 1.3 「每日」含义（必须写清）

- **种子**：`seed = f(本地日期 YYYY-MM-DD)`，例如字符串 `"2026-03-27"` 再哈希为 32bit 整数。  
- **同一日期、同一关卡索引**，任何设备上**迷宫拓扑一致**（若不做服务端，仅保证算法与种子一致即可）。  
- **换日 0:00**（本地）后种子变化，七关全部换新。  
- 可选：**挑战 ID** 显示为 `2026-03-27`，便于玩家分享「今日题」。

## 二、关卡与难度（指数级增长）

共 **7 关**。难度通过**迷宫尺寸**、**路径复杂度**、**视觉密度**递进；「指数」体现在**规模或分支量**随关卡近似按指数或幂次放大（在性能可接受前提下）。

### 2.1 推荐尺寸表（可微调）

| 关卡 | 建议逻辑格数（宽×高，奇数便于墙路） | 说明 |
|------|--------------------------------------|------|
| 1 | 9 × 9 | 入门，3～5 秒可走通 |
| 2 | 11 × 11 | |
| 3 | 15 × 15 | |
| 4 | 21 × 21 | |
| 5 | 29 × 29 | 移动端注意帧率 |
| 6 | 37 × 37 | 可降帧绘制或视口裁剪 |
| 7 | 45 × 45 或 **上限**（如 41×41） | 若卡顿则改为「同尺寸但增加岔路/死胡同比例」而非无限放大 |

**指数公式示例（实现时取整、设上下限）：**

- `side(level) = min( maxSide, floor( base * Math.pow(1.35, level - 1) ) | 奇数化 )`  
- 或固定上表 + 第 5～7 关在算法里提高 **随机回溯深度** / **死胡同比例**。

### 2.2 起点与终点

- 默认：**左上角格为起点**，**右下角格为终点**（或对称：左下→右上）。  
- 必须在**可通行区域**内，且迷宫生成后 **BFS 可达**（见下文）。

## 三、程序化迷宫算法

### 3.1 迷宫类型

- 使用 **完美迷宫（Perfect maze）**：任意两格之间**唯一路径**。  
- 优点：**最短步数**有明确定义（从起点到终点的最短路径长度唯一），便于校验、提示与排行榜公平性。  
- 实现任选其一：**递归回溯（Recursive Backtracker）**、**Randomized Kruskal**、**Randomized Prim**（均在网格上操作）。

### 3.2 随机性与种子

- 使用 **可复现的伪随机数生成器（PRNG）**，例如 **mulberry32** 或 **LCG**，初始化 `rng(seed + levelIndex * 常量)`。  
- 迷宫生成、墙壁打通顺序**全部**从该 RNG 取随机，保证**同日同关**结果一致。

### 3.3 生成后校验

- 对网格做 **BFS**（四连通），确认终点可达。  
- 计算 **理论最短步数** `optimalSteps`（BFS 层数或边数，按「每移动一格算一步」定义）。  
- 可选：通关时若 `playerSteps === optimalSteps`，给予「最短路径！」徽章。

## 四、步数规则与排行榜

### 4.1 步数定义

- **一步**：玩家从当前格移动到**正交相邻**的下一格（上下左右），且目标格为**通路**。  
- **不允许**斜走。  
- **退回已走格**也算一步（鼓励规划；若希望防刷步，可另做「有效步数」——规格可选，默认简单计步即可）。  
- **每关独立**记录 `stepsLevel[1..7]`；**总步数** `totalSteps = sum(stepsLevel)`。

### 4.2 排行榜（本地优先）

- 使用 **`localStorage`**（或 IndexedDB）存储，键建议含日期，例如 `mazeDailyRank_2026-03-27`。  
- 每条记录：`{ nickname, totalSteps, levelSteps[], timestamp, runId }`。  
- **排序**：`totalSteps` **升序**；同分可按 **timestamp 升序**（更早靠前）或 **更晚靠前**（按产品设计二选一）。  
- **保留条数**：如前 **20 名**；新记录插入后裁剪。  
- **同设备多昵称**：允许；防作弊仅提示「本地榜仅供娱乐」，不承诺防改。

### 4.3 可选扩展

- **全局榜**：需后端 + 用户鉴权，校验迷宫种子与提交路径（服务器重放 BFS 或验路径串），本提示词不强制。

## 五、像素风 UI / UX

### 5.1 视觉

- **画布**：低分辨率逻辑网格（如每格 8×8 或 16×16 CSS 像素），`image-rendering: pixelated`。  
- **墙**：深色方块 + 高光/阴影描边（可参考 PixelNoise 的 MC 方块 HUD）。  
- **路**：略浅底色或点阵纹理。  
- **玩家**：2×2 或单格精灵，小弹跳或踏步动画（2～4 帧循环）。  
- **出口**：不同色块 / 旗帜 / 传送门像素画。  
- **关卡标题**：`第 3 / 7 关 · 今日 2026-03-27`。  
- **步数 HUD**：本关步数 / 今日累计步数；可选显示「纪录：最优 xxx」。

### 5.2 交互

- **键盘**：方向键 / WASD。  
- **触摸**：滑动判定方向，或屏幕四角/虚拟方向键（大块像素按钮）。  
- **撤销一步**（可选）：降低挫败感，排行榜可注明「含撤销」或禁用撤销模式。  
- **暂停 / 返回**：确认对话框，防误触。

### 5.3 动效

- 通关：小范围**像素粒子**或**门开溶解**（与 PixelNoise 溶解类似可复用概念）。  
- 进入下一关：**切屏转场**（淡入或横向推屏）。

## 六、技术实现要点

### 6.1 技术栈

- **单页 HTML + CSS + JS** 即可；或 Vite 单页，部署 **GitHub Pages**。  
- 渲染：**Canvas 2D**（推荐，易缩放）或 **DOM + CSS Grid**（格数少时可行，大关卡性能注意）。

### 6.2 数据结构

- `maze[w][h]`：`0` 路、`1` 墙（或位掩码四向墙）。  
- `player = { x, y }`；移动前检测目标格。

### 6.3 状态机

- `menu` → `playing`（level 1..7）→ `summary`（展示七关步数与总步数）→ 写入榜 → `menu` / `replayToday`。

### 6.4 性能

- 第 6～7 关若格数 > 1000，**只绘制视口内格子**或降低重绘频率（玩家移动时再画）。  
- 生成迷宫在 **Web Worker** 中运行（可选），避免主线程卡顿。

## 七、验收标准（Checklist）

- [ ] 同一日期、同一关卡，刷新页面后迷宫**完全一致**。  
- [ ] 换日期后迷宫**变化**。  
- [ ] 七关尺寸或复杂度**明显递增**，第 7 关明显难于第 1 关。  
- [ ] 步数统计正确；**总步数**为七关之和。  
- [ ] 本地排行榜按总步数排序，可写入昵称。  
- [ ] 移动端可玩（触摸或虚拟键）。  
- [ ] 无控制台报错；可选：BFS 最短步与玩家步数对比展示。
